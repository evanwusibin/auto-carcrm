# 无答案与补充提问流程

> 流程编号：FLOW-02-03 | 版本：v1.1 | 更新时间：2026-06-13

**流程说明**：当检索结果置信度不足时，系统不应强行生成答案，而应进入追问、提示资料不足、或知识缺口记录流程。

---

## Typora 兼容版流程图

```mermaid
flowchart TD
    A[用户问题进入检索] --> B[六路检索]
    B --> C[RRF融合]
    C --> D[元数据过滤]
    D --> E[Rerank精排]
    E --> F[置信度计算]

    F --> G{是否达到回答阈值}
    G -- 是 --> H[进入正常回答流程]
    G -- 否 --> I[进入低置信度处理]

    I --> J{原因分析}
    J -- 缺少车型 --> K1[追问车型]
    J -- 缺少里程 --> K2[追问里程]
    J -- 缺少故障码 --> K3[追问故障码]
    J -- 问题不清晰 --> K4[追问更多现象]
    J -- 知识库无资料 --> K5[提示资料不足]
    J -- 权限不足 --> K6[提示无权限]

    K1 --> L[等待用户补充]
    K2 --> L
    K3 --> L
    K4 --> L
    L --> M{用户是否补充}
    M -- 是 --> A
    M -- 否 --> N[记录知识缺口]

    K5 --> N
    K6 --> O[记录权限日志]
```

## 当前代码对应关系

- 置信度判断：`app/rag/query/confidence_service.py`
- 答案生成：`app/rag/query/answer_service.py`
- Prompt 构建：`app/rag/query/prompt_builder.py`

---

## 当前追问策略可增强的方向

1. 追问原因分类更细
   - 当前主要基于分数和是否识别主体
   - 后续可以细分为车型缺失、文档类型缺失、案例信息缺失等

2. 追问话术更自然
   - 区分故障问题、保养问题、质保问题
   - 根据意图返回不同追问模板

3. 知识缺口记录更结构化
   - 记录缺口类别
   - 记录触发问题
   - 记录最大召回分数
   - 记录建议补充方向
