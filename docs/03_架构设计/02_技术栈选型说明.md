## 二、关键选型理由

### 2.1 为什么选 MongoDB？

**原因一：数据模型灵活**

RAG 系统中的知识 Chunk 元数据非常丰富且字段不固定。不同类型文档（保养手册/维修手册/案例）的 Chunk 元数据字段差异很大。MongoDB 文档模型天然支持这种灵活结构，无需为每类文档设计不同的表结构。

**原因二：向量与业务数据一体存储**

MongoDB Atlas Vector Search 支持在同一 Collection 中存储文本、向量和元数据，并支持向量检索时结合元数据过滤。这意味着不需要单独部署 Milvus 等独立向量数据库，降低了系统复杂度。

**原因三：适合嵌套复杂业务对象**

报修单、诊断会话等业务对象包含嵌套子结构（诊断结果、质保预判、维修历史），MongoDB 的嵌套文档比关系型数据库的多表 JOIN 更直观。

**选型对比：**

| 数据库 | 优势 | 劣势 | 本项目适合度 |
|---|---|---|---|
| MongoDB | 文档灵活，向量一体，嵌套支持 | 事务能力弱于关系型DB | ⭐⭐⭐⭐⭐ 推荐 |
| MySQL/PostgreSQL | 强事务，成熟稳定 | 向量需 pgvector，表结构固定 | ⭐⭐⭐ 适合第二阶段 |
| Milvus | 向量检索性能极强 | 独立部署，运维成本高 | ⭐⭐⭐ 适合大规模生产 |

### 2.2 为什么选 FastAPI？

- 原生支持异步（async/await），适合 RAG 系统中的多路并发检索
- 自动生成 OpenAPI 文档，降低前后端联调成本
- 类型注解 + Pydantic 数据校验，适合快速构建稳定 API
- 性能接近 NodeJS，远优于 Django/Flask

### 2.3 为什么选 bge-large-zh / bge-m3？

- BAAI 开源，中文语义理解能力强
- bge-large-zh：专为中文设计，适合商用车业务中文资料
- bge-m3：多语言版本，支持中英混合文档
- 在 MTEB 中文排行榜表现优秀
- 可本地部署，保证企业数据安全

### 2.4 为什么选 LangChain？

- 提供文档加载、切分、检索、链式调用等基础能力
- 快速原型验证：可快速接入不同 LLM 和向量库
- 活跃社区，持续更新，与 MongoDB 等有官方集成

### 2.5 为什么选 Qwen / DeepSeek？

- 国产大模型，中文理解能力强
- DeepSeek：推理能力强，API 价格低，适合生产部署
- Qwen2.5：指令遵循能力好，适合格式化输出需求
- 可本地部署，适合企业私有化部署需求
