## 三、核心知识点

### 3.1 导入流程知识点

```
1. 文件类型识别 → 路由开关
2. PDF解析 → MinerU（信任代理+轮询）
3. 图片增强 → Vision模型+MinIO上传
4. 文档切块 → 按标题切+长切短合
5. 主体识别 → LLM识别+file_title兜底
6. 向量化 → BGE-M3稠密+稀疏
7. 入库 → 幂等设计（先删后插）
8. 元数据抽取 → 车型/版本/有效期
9. 知识文档持久化 → MongoDB存储
10. 审核发布 → draft→published
```

### 3.2 查询流程知识点

```
1. 意图识别 → 6类意图分类
2. 实体抽取 → 车型/VIN/故障码
3. 向量检索 → BGE-M3混合搜索
4. HyDE检索 → 假设性答案+向量化
5. BM25检索 → 关键词精确匹配
6. 结构化查询 → MongoDB车辆/保养/质保
7. 案例检索 → MongoDB案例库
8. RRF融合 → 加权倒数排名融合
9. 重排序 → CrossEncoder精排
10. 元数据过滤 → 车型/版本/有效期
11. 置信度判断 → 追问机制
12. 答案生成 → Prompt+引用+置信度
13. QA落库 → MongoDB保存
```

### 3.3 核心概念

| 概念 | 一句话解释 | 面试怎么答 |
|------|-----------|-----------|
| **RAG** | 检索增强生成：先检索再生成 | "先从知识库检索相关文档，再让LLM基于文档生成答案" |
| **LangGraph** | 流程编排框架 | "用StateGraph定义节点和边，支持条件路由和并发执行" |
| **Milvus** | 向量数据库 | "存储稠密+稀疏向量，支持混合搜索" |
| **BGE-M3** | 向量化模型 | "生成1024维稠密向量+稀疏向量" |
| **RRF** | 融合算法 | "对多路检索结果按排名倒数加权求和" |
| **Rerank** | 重排序 | "用CrossEncoder对候选结果精细打分排序" |
| **HyDE** | 假设性检索 | "先让LLM生成假设性答案，用答案去检索" |
| **SSE** | 流式推送 | "服务端逐token推送给前端，实现打字机效果" |
| **幂等** | 重复执行安全 | "先删后插，重复导入不会产生重复数据" |

## 四、面试技巧

### 4.1 回答模板

**面试官**：介绍一下你的RAG项目？

**你**：
> 这是一个商用车智能售后知识库系统，技术栈是LangGraph+Milvus+BGE-M3+FastAPI。
>
> **导入流程**：PDF/MD → 解析 → 切块 → 向量化 → 入库。核心是幂等设计（先删后插）和图片增强（Vision模型+MinIO）。
>
> **查询流程**：问题 → 意图识别 → 实体抽取 → 多路检索（向量+HyDE+BM25） → RRF融合 → 重排序 → 元数据过滤 → 置信度判断 → 生成答案。
>
> **亮点**：支持6类意图分类、实体抽取、置信度追问、元数据过滤。

### 4.2 常见问题回答

**Q：为什么用Milvus不用Elasticsearch？**
> A：Milvus专门做向量检索，支持稠密+稀疏混合搜索，语义匹配能力更强。Elasticsearch擅长关键词检索，但语义理解弱。

**Q：RRF是什么？怎么实现的？**
> A：RRF是倒数排名融合算法。公式：score = 1/(k+rank)。k=60是平滑参数。对多路检索结果按排名倒数加权求和，排名越靠前分数越高。

**Q：为什么需要Rerank？**
> A：向量检索是粗筛，速度快但精度有限。Rerank用CrossEncoder对query和doc一起编码，精度更高，是精排。

**Q：HyDE是什么？**
> A：假设性文档嵌入。先让LLM生成假设性答案，用答案去检索，语义更丰富，召回率更高。

**Q：怎么保证导入不重复？**
> A：幂等设计。导入前先按item_name删除旧数据，再插入新数据。重复导入不会产生重复数据。
