### 2.2 背景轮播设计

- **图片数量**：4张（比亚迪 E6 / 汉 / 唐 / 海豹）
- **切换间隔**：4500ms，渐变淡入淡出（1.2s transition）
- **蒙层**：深蓝渐变 `rgba(10,30,80,.70) → rgba(10,30,80,.12)`，确保左侧文字可读
- **鼠标悬停**：暂停自动播放
- **轮播小点**：左下角，点击可跳转，当前项变为宽矩形

### 2.3 右侧卡片登录区设计

| 区域 | 说明 |
|------|------|
| 品牌行 | Logo + 平台名称，与左侧呼应，增强品牌一致性 |
| 标题区 | "欢迎登录"（23px/800粗）+ 副标题（灰色说明文字）|
| Tab切换 | 密码登录 / 验证码登录，带图标，带滑动背景高亮 |
| 输入框 | 带左侧图标，聚焦时主色边框 + 蓝色光晕 |
| 密码框 | 右侧眼睛图标切换明/暗文，点击图标切换 |
| 验证码 | 60秒倒计时，手机号格式验证（`/^1[3-9]\d{9}$/`）|
| 提交按钮 | 渐变蓝色，加载态 spinner，成功跳转 import.html |
| SSO | 企业单点登录按钮，样式区别于主按钮 |
| 安全提示 | 绿色底色，🔒图标，强调 HTTPS 加密 |

### 2.4 响应式断点

- `≤ 860px`：隐藏左侧品牌区，登录卡片全宽显示

### 3.2 核心交互流程

```
用户选择文件
     │
     ▼
前端预验证（格式/大小）
     │
     ▼
显示文件预览卡片（文件名/大小/类型选择下拉）
     │
     ▼
用户点击【开始导入】
     │
     ▼
POST /api/documents/upload
     │
     ├──▶ 后端：PDF/DOCX解析 → 文本切块 → 向量化 → 写入Milvus
     │
     ├──▶ 前端轮询进度（GET /api/documents/{id}/status）
     │         │
     │         ▼
     │    进度条更新（解析中 → 切块中 → 向量化中 → 完成）
     │
     ▼
导入完成 → 文档列表刷新 → 可进入检索页使用
```

### 3.3 数据处理流程

```
原始文档（PDF/DOCX）
     │
     ▼ OCR / 文本提取
文本内容
     │
     ▼ 智能切块（RecursiveTextSplitter，chunk_size=512，overlap=64）
知识片段（Chunks）
     │
     ├──▶ 向量化（EmbeddingModel）──▶ Milvus 向量库
     │
     ├──▶ 关键词索引 ──────────────▶ Elasticsearch
     │
     └──▶ 元数据提取 ──────────────▶ MySQL（文档表+片段表）
```

### 4.2 流式/非流式切换说明

| 模式 | 说明 | UI表现 |
|------|------|--------|
| **流式输出（Streaming）** | 边生成边展示，首字节延迟约300ms | 紫色开关激活，顶部RAG信息条显示，文字逐字输出带光标动画 |
| **非流式输出（Blocking）** | 生成完成后一次性返回 | 开关灰色，显示"思考中"loading动画，完整内容一次显示 |

**实现原理**：
```
流式模式（SSE）:
  POST /api/chat/stream
  Content-Type: text/event-stream
  
  data: {"type":"rag_step","step":"query_expand","status":"done"}
  data: {"type":"rag_step","step":"retrieve","status":"done"}
  data: {"type":"token","content":"根据"}
  data: {"type":"token","content":"T5"}
  data: {"type":"done","refs":[...]}

非流式模式：
  POST /api/chat
  Content-Type: application/json
  Response: {"answer":"...", "refs":[...], "stats":{...}}
```

## 5. 数据库设计（v1.2）

### 5.1 用户表 `users`

```sql
CREATE TABLE users (
    id           BIGINT       PRIMARY KEY AUTO_INCREMENT,
    username     VARCHAR(64)  NOT NULL UNIQUE COMMENT '用户名/工号',
    phone        VARCHAR(20)  UNIQUE COMMENT '手机号',
    password_hash VARCHAR(128) COMMENT 'bcrypt哈希',
    role         ENUM('admin','editor','viewer') DEFAULT 'viewer' COMMENT '角色',
    department   VARCHAR(64)  COMMENT '所属部门',
    real_name    VARCHAR(32)  COMMENT '真实姓名',
    sso_id       VARCHAR(128) COMMENT '企业SSO标识',
    last_login   DATETIME     COMMENT '最近登录时间',
    login_count  INT          DEFAULT 0 COMMENT '登录次数',
    is_active    TINYINT(1)   DEFAULT 1 COMMENT '是否启用',
    created_at   DATETIME     DEFAULT CURRENT_TIMESTAMP,
    updated_at   DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_phone (phone),
    INDEX idx_sso_id (sso_id)
) COMMENT '用户表';
```

### 5.2 知识文档表 `documents`

```sql
CREATE TABLE documents (
    id           BIGINT       PRIMARY KEY AUTO_INCREMENT,
    title        VARCHAR(256) NOT NULL COMMENT '文档标题',
    filename     VARCHAR(256) COMMENT '原始文件名',
    file_path    VARCHAR(512) COMMENT '存储路径',
    file_size    BIGINT       COMMENT '文件大小(字节)',
    doc_type     ENUM('maintenance','warranty','repair','fault_code','case','other')
                              NOT NULL COMMENT '文档类型',
    vehicle_model VARCHAR(64) COMMENT '适用车型（如T5轻卡）',
    version      VARCHAR(32)  COMMENT '文档版本',
    status       ENUM('pending','processing','completed','failed')
                              DEFAULT 'pending' COMMENT '处理状态',
    chunk_count  INT          DEFAULT 0 COMMENT '知识片段数量',
    vector_status TINYINT(1)  DEFAULT 0 COMMENT '是否已向量化',
    error_msg    TEXT         COMMENT '错误信息',
    uploaded_by  BIGINT       COMMENT '上传用户ID',
    published_at DATETIME     COMMENT '发布时间',
    created_at   DATETIME     DEFAULT CURRENT_TIMESTAMP,
    updated_at   DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_doc_type (doc_type),
    INDEX idx_status (status),
    INDEX idx_vehicle_model (vehicle_model),
    FOREIGN KEY (uploaded_by) REFERENCES users(id)
) COMMENT '知识文档表';
```

### 5.3 知识片段表 `document_chunks`

```sql
CREATE TABLE document_chunks (
    id           BIGINT       PRIMARY KEY AUTO_INCREMENT,
    document_id  BIGINT       NOT NULL COMMENT '所属文档ID',
    chunk_index  INT          NOT NULL COMMENT '片段序号',
    content      MEDIUMTEXT   NOT NULL COMMENT '片段文本内容',
    page_num     INT          COMMENT '原始页码',
    section      VARCHAR(256) COMMENT '章节标题',
    token_count  INT          COMMENT 'Token数量',
    vector_id    VARCHAR(128) COMMENT 'Milvus向量ID',
    created_at   DATETIME     DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_document_id (document_id),
    FOREIGN KEY (document_id) REFERENCES documents(id) ON DELETE CASCADE
) COMMENT '知识片段表';
```

### 5.4 会话表 `chat_sessions`

```sql
CREATE TABLE chat_sessions (
    id           BIGINT       PRIMARY KEY AUTO_INCREMENT,
    session_uuid VARCHAR(64)  NOT NULL UNIQUE COMMENT '会话UUID',
    user_id      BIGINT       COMMENT '用户ID（匿名则为NULL）',
```

## 6. RAG 核心流程说明

### 6.1 检索增强生成（RAG）完整流程

```
Step 1: 问题理解
   用户输入 ──▶ 意图识别（保养/质保/故障/案例）
              ──▶ 实体提取（车型/故障码/里程等）

Step 2: 查询扩展（Query Expansion）
   口语表达 ──▶ 专业术语映射
   "挂不上挡" ──▶ "变速器故障/READY状态异常/整车控制器CAN通信"

Step 3: 四路并行检索
   Route A: 向量检索 ──▶ Embedding模型 ──▶ Milvus Top-20
   Route B: 关键词检索 ──▶ BM25 ──▶ Elasticsearch Top-20
   Route C: 结构化查询 ──▶ 质保周期/保养里程 ──▶ MySQL
   Route D: 案例检索 ──▶ 历史案例特征匹配 ──▶ Case DB

Step 4: 结果融合 & ReRank
   合并去重 ──▶ 交叉编码器（Cross-Encoder）重排序 ──▶ Top-5

Step 5: LLM生成
   构建Prompt：System提示 + 检索上下文 + 用户问题
   调用LLM ──▶ 流式（SSE）或非流式（HTTP JSON）

Step 6: 引用注入 & 风险评估
   标注来源文档 + 页码 + 相关度分数
   识别涉及高压/安全的内容 ──▶ 添加风险等级标注
```

### 6.2 多轮对话上下文管理

```
会话历史（Redis TTL=2h）
     │
     ▼
上下文窗口（最近4轮对话 + 当前问题）
     │
     ▼
若上下文超长（>2000 tokens）：
  ──▶ 摘要压缩（LLM生成历史摘要）+ 当前问题
```
